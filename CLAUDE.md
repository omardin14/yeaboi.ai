# CLAUDE.md

Guidance for Claude Code (and humans) working **on** yeaboi.ai.

## Stack

Rust (**edition 2024**) Cargo workspace + a **Tauri v2** desktop app (**React 19 + Vite +
Tailwind v4**). The Rust engine is presentation-agnostic and also ships a headless **`yeaboi` CLI**, so
the same `Snapshot` powers both the app and `yeaboi --json`. Rust↔TS types are generated with **ts-rs**.
No database. macOS-first (cross-platform later).

## Commands

You only need two; both set up their own deps/PATH:

| Command | What it does |
|---|---|
| `make dev` | run the desktop app (installs deps, builds, hot-reload, opens the window) |
| `make cli` | headless — print a live snapshot as JSON |
| `make test` | `cargo test --workspace` + frontend `vitest` |
| `make lint` | `rustfmt --check` + `clippy -D warnings` + `tsc` |
| `make verify` | everything CI runs (lint + test + cli) |
| `make gen-bindings` | regenerate the Rust→TS bindings |

Run `make` with no args for the list. (`cargo` lives at `~/.cargo/bin`; the Makefile adds it to PATH.)

## Where things live

```
crates/
  yb-core      domain model — the `Snapshot` contract + collectors.  NO UI, NO OS-specific code.
  yb-proc      OS introspection: process table, ports, signals.
  yb-exec      external command runner (spawn/stream/detached).
  yb-git       typed git/gh wrappers.
  yb-worktree  worktree engine (decentralized; MD5 ports).
  yb-agent     agent providers (claude/codex) + review orchestrator.
  yb-cli       headless `yeaboi` binary (--json/--once).
desktop/        Tauri app — src-tauri (Rust shell: commands + events) + src (React frontend).
  src/lib/bindings/   GENERATED by ts-rs — never hand-edit; run `make gen-bindings`.
```

- `yb-core::Snapshot` is the single contract the CLI **and** the desktop app render from.
- Frontend ↔ backend is **Tauri commands** (FE→Rust) + **events** (Rust→FE; the snapshot stream).

## Conventions (non-obvious — these are what reviewers enforce)

- **`yb-core` stays presentation-agnostic**: no `tauri`/`ratatui`/UI deps, and no OS-specific code
  (that belongs in `yb-proc`). **Nothing depends on a UI crate.**
- **`clippy -D warnings` is enforced.** No `#[allow(clippy::…)]` without a one-line justification.
- **No `unwrap()` / `expect()` / `panic!` / `todo!` / `unimplemented!` in runtime paths.** Return
  `Result` (errors via `thiserror`). `unwrap`/`expect` are fine in tests and at provable invariants
  (with a comment).
- **Never swallow errors.** No `let _ = fallible()`, empty `if let Err(_) {}`, `.ok()` that drops a
  real failure, or `unwrap_or_default()` that hides one. Surface errors to the UI as toasts; log with
  context. (Silent fallback is only OK when it is the documented happy path.)
- **Regenerate bindings** after changing any `#[ts]` model type: `make gen-bindings`. CI fails if
  `desktop/src/lib/bindings/` is stale. The `ts` feature on `yb-core` is for binding generation
  only — the CLI never enables it.
- **Ports** are deterministic `MD5(path)` (byte-compatible with reference repos); see `yb-worktree`.

## Parallel development (worktrees)

```
./scripts/w.sh <name>         # create ../ai-manager-<name> on branch <name>
./scripts/w.sh <name> open    # create + launch claude inside it
./scripts/w.sh <name> rm      # remove worktree + branch
```

Each worktree gets a **deterministic dev-server port** (the main checkout is 1420; worktrees hash to
1430–1529), so you can run `make dev` in several worktrees at once without collisions. Point a
worktree session at one task; review it from a *different* session.

## Review workflow

- `/review-pr` — fresh-context, **parallel** specialized reviewers on the current PR diff
  (`code-reviewer` always; `silent-failure-hunter` if error handling changed; `pr-test-analyzer` if
  tests changed; then `code-simplifier`). Returns Critical / Important / Suggestions / Verdict.
- `/cross-review` — layers **Codex** (GPT) on top of Claude's review to surface blind spots
  (two-phase; needs the codex plugin, enabled in `.claude/settings.json`).
- **Reviewer ≠ implementer:** run reviews in a fresh session so the reviewer isn't primed by the
  implementation chat. When a bug slips past review, add a rule here or sharpen the relevant agent.
